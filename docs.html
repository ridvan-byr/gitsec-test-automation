KILAVUZ

Genel Bakış

GitSec Test Otomasyonu, GitSec SaaS platformunun tüm entegrasyonlarını, kimlik doğrulama süreçlerini, yedekleme planlayıcılarını ve geri yükleme mekanizmalarını doğrulamak üzere geliştirilmiş yüksek performanslı ve modüler bir uçtan uca (E2E) test otomasyon projesidir.

Proje, web arayüzünü gerçek bir kullanıcı gibi simüle ederek tarayıcı üzerinde koşturmak için modern Playwright kütüphanesini ve tip güvenliği için TypeScript dilini temel alır.

Hız & Kararlılık

Paralel koşum desteği, otomatik bekleme (auto-wait) ve akıllı yeniden deneme (retry) mekanizmalarıyla kararlı çalışma.

Gelişmiş Denetim

Test adımları sürerken arka planda oluşan tüm ağ istekleri ve konsol hataları anlık olarak taranır.

Çevre Editörü

Yerel arayüz üzerinden .env dosyasını canlı olarak güncelleyebilir ve testlerinizi özelleştirebilirsiniz.

Bilgi: Bu otomasyon paketi, Staging ortamında gerçek zamanlı entegrasyonları test eder. Güvenli API bağlantıları ve sahte (mock) sağlayıcı yapılandırmalarıyla izole edilmiş test ortamı sağlar.
TASARIM ŞABLONU

Mimari & Page Object Model (POM)

Kod tekrarını önlemek, seçicileri (selectors) tek bir noktada toplamak ve arayüz değişikliklerinde testlerin kolayca güncellenebilmesini sağlamak amacıyla projede Page Object Model (POM) mimarisi uygulanmıştır.

Tüm sayfa nesneleri tests/pages/ dizininde yer alır. Temel sayfa sınıfları şunlardır:

  • LoginPage: GitSec giriş, kayıt ve şifre sıfırlama adımlarının elementlerini yönetir.
  • ProviderPage: GitHub, GitLab ve Bitbucket gibi kod sağlayıcı kartlarının bağlantı durumlarını denetler.
  • StoragePage: OneDrive, Google Drive ve Huawei OBS gibi bulut depolama sağlayıcılarının form entegrasyonunu yönetir.
  • RestorePage: Geri yükleme sihirbazındaki (Restore Wizard) tüm adımları, dropdown seçimlerini ve tetikleme butonlarını kapsar.

Örnek bir POM kullanımı şu şekildedir:

import { test } from '../../fixtures/test';

test('Yedekleme ve Geri Yükleme Akışı', async ({ page, restorePage }) => {
  await restorePage.completeTargetOrganizationStep(async () => {
    // Popup yetkilendirmesi tetiklenir
  });
  await restorePage.selectBackupSourceIfVisible();
});

├── pages/ // Page Object Model (POM) sınıfları
│ ├── LoginPage.ts
│ ├── ProviderPage.ts
│ ├── RestorePage.ts
│ └── StoragePage.ts
└── e2e/ // Test paketleri (Test suites)
├── auth/ // Giriş ve kimlik doğrulama testleri
├── backups/ // Zamanlayıcılar ve geri yükleme testleri

Öneri: Sayfa nesneleri (POM) `data-testid` veya erişilebilirlik (ARIA) rollerine göre seçilmiştir. Bu sayede UI değişikliklerinde testlerin kırılganlığı (flakiness) en aza indirilmiştir.
KAPSAM

Test Kapsamı

Otomasyon paketi, GitSec platformunun kritik yollarını (critical paths) ve hata sınırlarını test eden kapsamlı senaryolardan oluşur:

1. Backup Schedulers (Zamanlayıcılar)

Kullanıcıların depoları için otomatik yedekleme planları oluşturma süreçlerini test eder. Günlük (Daily), Haftalık (Weekly) ve Aylık (Monthly) scheduler formlarının doldurulması, timezone seçimleri, listenin kontrol edilmesi ve ardından temizlik amacıyla planın silinmesi akışlarını doğrular.

2. Storage Integration (Depolama Entegrasyonu)

Google Drive, OneDrive ve Huawei OBS sağlayıcılarının bağlantı adımlarını doğrular. Doğru kimlik bilgileriyle başarılı bağlantı senaryolarının yanı sıra, geçersiz anahtar/şifre durumlarında backend'den dönen hata mesajlarının arayüzde doğru yansıtıldığını kontrol eder.

3. Restores (Geri Yükleme Sihirbazı)

Platformun en hassas bileşenlerinden biri olan yedek geri yükleme süreçlerini kapsar. Adımlar arasında geri/ileri gidildiğinde form verilerinin korunup korunmadığını, mükerrer repo isimleri veya geçersiz karakterler girildiğinde backend validation kurallarının tetiklendiğini teyit eder.

GÜVENLİK VE DENETİM

Arka Plan Hata Denetimi (Audit)

Platform arayüzünde kullanıcıya hata yansıtılmasa bile arka planda (console loglarında veya network tabında) sessizce çöken hataları tespit etmek amacıyla Gelişmiş Denetim Modülü geliştirilmiştir.

Bu modül tests/fixtures/test.ts dosyası içinde yer alır ve her test sırasında tarayıcı olaylarını dinler:

  • JSException: Sayfada yakalanamayan JavaScript hatalarını (uncaught exceptions) yakalar.
  • ConsoleError: Geliştiricilerin konsola düştüğü hata loglarını (console.error) izler.
  • NetworkFailure: HTTP statüsü 400 ve üzerinde olan tüm başarısız API isteklerini denetler.

Hata İstisnaları (ignoredErrors)

Bazı test senaryoları doğası gereği hata tetiklemek üzere yazılmıştır (Örn: mükerrer geri yüklemede 409 Conflict alma testi). Bu gibi durumlarda, testin hatalı sonuçlanmaması için ilgili hata kalıpları page.ignoredErrors listesine eklenerek denetimden muaf tutulur: 

// 409 Conflict hatasını denetim dışı bırak
(page as any).ignoredErrors = [/restore\/schedules\/trigger/, /status of 409/];
Dikkat: Staging veritabanı Unicode (SQL_ASCII) formatında yapılandırılmışsa, testlerde Türkçe karakter (ç, ı, ş, ğ, ö, ü) içeren veriler kullanılmamalıdır. Aksi halde veri tabanı kayıt sırasında Unicode hatası üretecektir.

Çalıştırma Rehberi

Testleri yerel dashboard üzerinden tek tıkla çalıştırabileceğiniz gibi, terminal üzerinden aşağıdaki komutları kullanarak da tetikleyebilirsiniz:

Çevre Değişkenleri Konfigürasyonu

Testlerin doğru şekilde yetkilendirilmesi ve çalışması için proje kök dizinindeki .env dosyasının düzenlenmesi gerekir. Arayüzdeki .env Yapılandırması sekmesini kullanarak bu değişkenleri şifreli ve kolayca düzenleyebilirsiniz:

  • E2E_USER_EMAIL & E2E_USER_PASSWORD: GitSec Staging hesabına giriş için kullanılan bilgiler.
  • GITHUB_TEST_USER & GITHUB_TEST_PASSWORD: GitHub kurulum ve yetkilendirme akışlarını doğrulamak için kullanılan test hesabı bilgileri.
  • WORKSPACE_ID: Testlerin yürütüleceği hedef GitSec workspace numarası (Örn: 28).
GÜVENLİK VE OTURUM

Kimlik Doğrulama & Oturum Yönetimi

Uçtan uca test otomasyonlarında her test dosyası veya senaryosu için sıfırdan kullanıcı girişi (Sign In) yapmak, hem zaman maliyetini artırır hem de gereksiz ağ trafiği ile testlerin kararlılığını riske atar. Bu sorunu önlemek için projede Playwright Session Storage Sharing mekanizması uygulanmıştır.

Oturum Paylaşım Mekanizması

Süreç, kimlik doğrulama adımlarının tek bir noktada toplandığı tests/e2e/auth.setup.ts dosyasında başlar. Bu dosya, belirlenen test kullanıcısı bilgileriyle GitSec API'sine bir HTTP `POST` isteği gönderir, dönen oturum token ve çerezlerini yakalar ve bunları diskteki güvenli bir JSON dosyasında depolar: 

// auth.setup.ts - Oturumu Kaydetme
import { test as setup } from '../fixtures/test';

const authFile = 'playwright/.auth/user.json';

setup('Giriş yap ve oturumu kaydet', async ({ page }) => {
  // Giriş API isteğini gönder ve token'ı al
  const response = await page.request.post('https://staging.api.gitsec.io/auth/signin', {
    data: { email: process.env.E2E_USER_EMAIL, password: process.env.E2E_USER_PASSWORD }
  });
  
  // Oturum durumunu (Cookies & LocalStorage) diske kaydet
  await page.context().storageState({ path: authFile });
});
Bilgi: Diske kaydedilen `user.json` dosyası, Playwright yapılandırmasında (`playwright.config.ts`) tanımlanan `storageState` parametresi aracılığıyla tüm test dosyalarına otomatik olarak enjekte edilir. Böylece testlerin tamamı oturum açmış olarak başlar.
OTOMASYON HATLARI

CI/CD Entegrasyonu (GitHub Actions)

Yazılan testlerin her kod değişikliğinde (push/pull request) veya düzenli aralıklarla (cron schedules) otomatik olarak çalışması için **GitHub Actions CI/CD Pipeline** entegrasyonu kurulmuştur.

Tüm boru hattı (pipeline) adımları .github/workflows/playwright.yml dosyasında tanımlıdır: 

# playwright.yml - GitHub Workflow
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  schedule:
    - cron: '0 0 * * 1' # Her Pazartesi gece yarısı çalıştır

jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: 18
        
    - name: Bağımlılıkları Yükle
      run: npm ci
      
    - name: Playwright Tarayıcılarını Yükle
      run: npx playwright install --with-deps
      
    - name: Testleri Koştur
      run: npx playwright test
      env:
        E2E_USER_EMAIL: ${{ secrets.E2E_USER_EMAIL }}
        E2E_USER_PASSWORD: ${{ secrets.E2E_USER_PASSWORD }}
Öneri: Pipeline sonunda oluşan test hata kayıtları, ekran görüntüleri ve videolar `playwright-report/` dizininde toplanarak GitHub Actions arayüzünde indirilebilir **Artifact** olarak saklanır.
KARARLILIK

Kararsız Testler & Akıllı Bekleme

Ağ gecikmeleri, sunucu yanıt süreleri veya asenkron render işlemleri test otomasyonlarında sahte hata (flakiness) oluşmasının en büyük nedenidir. Playwright, bu kararsızlıkları önlemek için yerleşik **Auto-Waiting (Otomatik Bekleme)** ve **Smart Assertions** mekanizmaları barındırır.

Bunun yanı sıra, ağ isteklerinin tamamlanmasını beklemek veya element durumlarını kontrol etmek için aşağıdaki bekleme stratejileri tercih edilir:

  • Locators auto-wait: `click()`, `fill()` gibi tüm tarayıcı komutları, ilgili element DOM'da belirene, görünür olana ve tıklanabilir hale gelene kadar (varsayılan 30s) otomatik olarak bekler.
  • Ağ İsteklerini Dinleme (waitForResponse): API isteklerinin tamamlandığından emin olmak için dinamik bekleme:
// API cevabını bekleme örneği
const responsePromise = page.waitForResponse(response => 
  response.url().includes('/api/v1/backups') && response.status() === 200
);
await page.getByRole('button', { name: 'Yedekle' }).click();
await responsePromise;
Önemli: Test kodlarında kesinlikle statik `page.waitForTimeout(milliseconds)` (hardcoded sleep) kullanılmamalıdır. Bu tür statik beklemeler test süresini gereksiz uzatır ve ağ yavaşladığında testi kırar.
REHBER

Yeni Test Ekleme Kılavuzu

Projeye yeni bir test senaryosu eklemek için aşağıdaki adımları sırasıyla takip etmeniz önerilir. Bu akış, projenin mevcut Page Object Model (POM) ve Audit (Denetim) mimarisine tam uyumluluk sağlar:

Adım 1: Page Object Sınıfını Kontrol Edin / Oluşturun

Eğer test edeceğiniz sayfa yeni bir sayfa ise, elementlerin seçicilerini ve yardımcı fonksiyonları tutacak bir POM sınıfı (`tests/pages/YeniSayfa.ts`) oluşturun.

Adım 2: Fixture Tanımını Güncelleyin

Oluşturduğunuz yeni POM sınıfını tests/fixtures/test.ts dosyasına ekleyin. Böylece test dosyalarında bu sayfaya doğrudan erişebilirsiniz.

Adım 3: Test Dosyasını Yazın

tests/e2e/ dizininde ilgili kategori altında yeni bir `.spec.ts` dosyası oluşturun: 

// tests/e2e/backups/yeni-ozellik.spec.ts
import { test } from '../../fixtures/test';
import { expect } from '@playwright/test';

test.describe('Yeni Özellik Test Paketi', () => {
  test('Başarılı Senaryo', async ({ page, loginPage }) => {
    // 1. Sayfaya yönlen (otomatik oturum açılmıştır)
    await page.goto('/dashboard');
    
    // 2. POM yardımıyla form doldur
    await loginPage.doSomething();
    
    // 3. Doğrulamaları yap
    await expect(page.locator('.status-badge')).toBeVisible();
  });
});
Denetim Uyumu: Yazdığınız test, custom `test` fixture'ını import ettiği için arka plan denetim modülü (Audit) otomatik olarak aktifleşir ve konsol/ağ hatalarını arka planda sizin yerinize denetler.
